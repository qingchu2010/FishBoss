# 工作流界面与 Bash 乱码处理说明

## 目的

这份文档给后续接手的模型或开发者使用，重点说明本轮改动的设计目标、代码位置、验证方式，以及 Bash 输出乱码的处理思路。

## 工作流界面改造

### 改造目标

旧界面把工作流基础信息、画布、节点属性、运行历史放在同一条长页面里，用户需要频繁滚动，并且“添加节点”“编辑属性”“运行查看”之间的关系不够直接。

新界面改成工作台结构：

- 左侧：工作流列表和新建入口。
- 中间：画布和节点工具栏，专注处理 DAG 拖拽、连线和状态观察。
- 右侧：属性与运行两个标签页，选中节点/连线时编辑属性，没有选中对象时编辑工作流基本信息。
- 顶部：保存、运行、删除、重置视图等高频操作集中放置。

### 主要文件

- `frontend/src/views/WorkflowsView.vue`
- `frontend/src/services/workflows.ts`
- `frontend/src/i18n/zh.ts`
- `frontend/src/i18n/en.ts`

### 行为变化

- 新增右侧 `属性 / 运行` 标签页，减少页面纵向滚动。
- 节点添加改成左侧画布工具栏，使用 lucide 图标。
- 运行 payload 和连线 metadata 增加 JSON 对象校验，非法时不提交。
- 运行历史移入右侧面板，点击节点运行状态可以同步选中画布节点。
- `canvasToWorkflow` 保留已有连线 metadata，避免用户编辑后保存丢失。
- `createWorkflowNodeId` 增加随机后缀，降低快速添加节点时 ID 碰撞概率。

### 商业级体验补强

- 保存草稿与运行校验分离：用户可以保存未完成工作流，避免丢草稿；运行前必须通过图结构校验。
- 未保存保护：切换工作流、离开页面、套用模板前会提示确认，减少误丢修改。
- 自动布局：顶部提供自动布局按钮，按起点和连线层级整理节点位置。
- 快速模板：提供智能体回复、工具后接智能体、条件分支归档三种起步模板。
- Agent 和 Tool 不再要求手填 ID：前端从后端 `GET /api/agents` 和 `GET /api/tools` 加载可选项，通过全局下拉框选择。
- 图结构质量栏：顶部展示阻塞问题和提醒，点击问题会定位到对应节点或连线。
- 运行保护：存在未保存修改或阻塞问题时，运行入口禁用并在运行面板展示原因。
- 工具参数表单化：工具节点选择工具后，根据后端工具注册表参数生成字段；字符串、数字、布尔值、枚举、数组、对象分别用合适控件编辑，同时保留完整 JSON 输入作为高级入口。
- 编辑历史：支持撤销、重做、复制节点、粘贴节点，并提供按钮与键盘快捷操作。

### 仍可继续打磨

- 当前没有引入自动截图测试，视觉质量主要依赖人工浏览器检查。
- 工具参数表单目前覆盖注册表参数类型，但还没有做每个工具的业务级解释、示例值或参数依赖关系。
- 复杂 DAG 的自动布局是轻量层级布局，不是完整图布局引擎；如果流程非常复杂，后续可引入专门布局算法。
- 撤销/重做基于前端快照，不涉及后端版本历史；后续如果要做审计级版本管理，应落到后端 WorkflowDefinition 版本。

## Bash 乱码处理

### 根因判断

Windows 下 `spawn(..., { shell: true })` 默认走 `cmd.exe`，输出编码通常跟当前代码页有关。后端之前直接用 `chunk.toString()` 按 UTF-8 解码 stdout/stderr，如果子进程实际输出 GBK/GB18030，中文会变成乱码。

另外，PowerShell 或终端本身也可能把 UTF-8 文件内容按本地代码页展示，导致看起来像文件损坏。用 Node 读取同一文件时能看到正常中文，说明文件本身是 UTF-8，问题在显示/解码链路。

### 本轮处理

文件：`src/server/domains/tools/tools/BashTool.ts`

处理方式：

- Windows 平台改为显式执行 `cmd.exe /d /s /c "chcp 65001 > nul & <command>"`，先把当前命令进程切到 UTF-8 代码页。
- 子进程环境增加 `PYTHONIOENCODING=utf-8`、`PYTHONUTF8=1`、`LANG=C.UTF-8`、`LC_ALL=C.UTF-8`，让常见运行时更倾向输出 UTF-8。
- stdout/stderr 先累积为 Buffer，再统一解码。
- UTF-8 解码结果出现替换字符时，回退用 `gb18030` 解码，兼容仍然输出 GBK 的 Windows 程序。

### 后续注意

如果某个外部程序强制按本地代码页输出，仍可能需要针对该程序设置环境变量或参数。优先在后端 Bash 工具内统一处理，不要在前端做乱码替换，因为前端无法可靠判断原始字节编码。

## 给 GPT-5.4 的接手提示

优先检查 `WorkflowsView.vue` 的实际浏览器效果，特别是宽屏、窄屏、拖拽节点、保存后重新加载、模板套用、自动布局、运行记录切换、未保存确认、工具参数表单、撤销重做、节点复制粘贴和右侧面板切换。若要继续优化，下一步建议做工具参数示例、模板市场、节点搜索、节点批量选择和完整截图回归测试。

Bash 乱码如果仍出现，先确认乱码来自后端工具返回值，还是来自当前终端显示。后端返回值问题继续查 `BashTool.ts`；终端显示问题优先检查 PowerShell/Windows Terminal 的编码设置。
